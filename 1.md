...一、需求理解

我想做一个独立的订阅前置网关，本质是「反爬虫/反扫描的应用层 WAF」，专门保护 V2Board 的订阅接口。
[用户/扫描器]  →  [前置网关 ⭐ 我们要做的]  →  [V2Board 后端]
                      ↓
                 [独立日志/封禁库]
...核心是 拦截 + 记录 + 决策，但不碰 V2Board 数据库，保持解耦。

二、需求拆解

2.1 数据采集维度（每次请求记录）
- Token（订阅链接里的 token，作为用户标识）
- 请求 IP（注意 X-Forwarded-For、CF-Connecting-IP）
- User-Agent（识别 Clash/Shadowrocket/V2RayN/curl/python-requests 等）
- 请求时间戳
- 请求路径（/api/v1/client/subscribe 等）
- 响应状态（正常/封禁/虚假）
- 可选：ASN、国家、TLS 指纹（JA3/JA4）

2.2 异常判定规则（多维度）
频率

• 维度: 频率

• 异常特征: 单 token 短时间高频拉取

• 阈值参考: 1分钟>10次 / 1小时>60次

IP 分散

• 维度: IP 分散

• 异常特征: 同 token 多 IP 并发

• 阈值参考: 5分钟内 >5 个不同 IP

UA 异常

• 维度: UA 异常

• 异常特征: curl/python/go-http-client/空 UA

• 阈值参考: 黑名单匹配

时间分布

• 维度: 时间分布

• 异常特征: 完美等间隔（机器人特征）

• 阈值参考: 标准差<X

ASN

• 维度: ASN

• 异常特征: 来自数据中心/扫描器ASN

• 阈值参考: IP2Location/ipinfo

路径扫描

• 维度: 路径扫描

• 异常特征: 同 IP 触发多个不存在 token

• 阈值参考: >3次404

2.3 响应策略（分级处置）
1. 正常 → 透传到 V2Board
2. 可疑（黄） → 限速 / 增加延迟 / 记录观察
3. 异常（橙） → 返回虚假节点（关键功能⭐）
4. 恶意（红） → 直接 403 / 加入封禁列表 / 推送到防火墙

2.4 虚假节点生成
- 随机生成看似合法的 SS/VMess/Trojan 配置
- 服务器指向 黑洞 IP（如 127.0.0.1、0.0.0.0、或专门的蜜罐）
- 让扫描者以为拿到了真订阅，浪费对方时间/资源
- 可以混入真节点格式让攻击者难以分辨

三、技术选型建议

方案 A：Go 单体（推荐⭐）
- 优势：性能强、单文件部署、并发好、内存小
- 栈：Go + Fiber/Gin + SQLite/BoltDB + Redis（限频）
- 适合：主人有 13 个落地节点，资源敏感

方案 B：Python FastAPI
- 优势：开发快、改规则容易、生态好
- 栈：FastAPI + SQLite + Redis
- 适合：规则迭代频繁、需要 ML 异常检测

方案 C：Nginx + Lua (OpenResty)
- 优势：直接在网关层处理，零额外进程
- 劣势：调试麻烦、Lua 写复杂逻辑痛苦
- 适合：纯规则引擎，无需复杂决策

方案 D：Caddy + 自写插件
- 优势：自动 HTTPS、配置简洁
- 劣势：Go 插件需要重新编译 Caddy

大福推荐方案 A（Go），原因：
- 部署简单（一个二进制扔上去）
- 性能足够扛住扫描洪水
- SQLite 嵌入式存储，无外部依赖

五、关键设计决策点（需要主人确认）

主人决定以下几点，大福才好动手：

Q1. 部署位置
- (a) 和 V2Board 同机器（Nginx 反代到本地端口）
- (b) 独立机器（前置一台专门防扫描的网关）
- (c) Cloudflare Worker 前置 + 自建后端兜底

Q2. 数据库
- (a) SQLite（轻量、单文件、够用）
- (b) MySQL/PostgreSQL（已有 V2Board 的库可共用实例但独立 DB）
- (c) ClickHouse（日志量大、查询分析强，但重）

Q3. 限频存储
- (a) Redis（推荐，主人已经在用）
- (b) 内存（重启丢失，但极快）
- (c) SQLite + 内存缓存

Q4. 虚假节点策略
- (a) 完全随机生成（最简单）
- (b) 真实格式 + 黑洞 IP（迷惑性强）
- (c) 蜜罐节点（攻击者连上后还能反向追踪）

Q5. 管理界面
- (a) 命令行 + 配置文件（极简）
- (b) 简单 Web UI（看日志、改规则）
- (c) 对接现有面板（Grafana/自建）

Q6. 告警通道
- (a) 不告警，只记录
- (b) Telegram Bot 推送（主人已有 Hermes）
- (c) 邮件 / Webhook

Q7. 多面板支持
有 二狗加速 + 快喵加速 两个品牌，是否：
- (a) 一个网关同时保护两个面板（按域名路由）
- (b) 每个面板独立部署一个网关

六、可能的坑（提前预警）

1. CDN 透传：如果前面有 Cloudflare，要正确解析真实 IP（CF-Connecting-IP）
2. 订阅缓存：客户端（Clash）会自动定时拉，正常用户也可能"高频"
3. 多端登录：同一 token 多设备订阅是合法行为，别误杀
4. 虚假节点反噬：如果生成的假节点格式有缺陷，反而暴露你的网关
5. token 泄露：日志里存明文 token 风险高，建议存 hash(token)
6. GDPR/隐私：记录 IP+UA 涉及用户隐私，建议加保留期限
